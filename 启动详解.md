# IND Knowledge 系统启动与交付详解

这份文档将详细为您介绍当前 RAG 系统的架构、运行准备以及一键启停等交付说明。该系统已经被重构为**基于容器技术的完整交付环境包**，这保证了其无论在什么端都能拥有一致的环境表现。

## 1. 系统架构速览

系统基于 Docker Compose，将三个核心微服务隔离封装：

- **🟢 数据库层 (db)**: 部署了正宗的 `Neo4j:5` 镜像实例。承载由后端同步过来的图谱事实三元组，原生支持高效率的图形拓展检索检索。
- **🔵 后端引擎层 (backend)**: 基于 `Python 3.10` 的 FastAPI / RAG 系统。内部集成了混合检索和问答链路。它在容器每次拉起时，都会**自动检测增量 Markdown 并在进入运行挂起前自动进行向量、图和关键字等存储落盘导入引擎库**。
- **🟡 前端展现层 (frontend)**: 包含经过 Node.js 环境彻底打包的 Vite + React 静态产出物。容器使用 `Nginx` 反向代理进行直接发布，并在内部将您的所有 `/api/*` 请求无缝直接转发给后端的应用端口。

## 2. 部署前置条件

由于彻底落实了容器化设计，系统对于宿主机的系统侵入降到了最低，只需：
1. 操作系统：Windows (WLS2 / Hyper-V)、macOS（Silicon/Intel）或 Linux。
2. 安装环境：**仅需安装 Docker 与 Docker Desktop**，并确认启动即可。无需在宿主机环境中配备任何 Node 或 Python 版本。 
3. **环境密钥 (`.env` 文件配置)**：首次获取部署包时，请依据根目录下 `.env.example`，在同级目录复制为 `.env` 文件。填入必须的各项大模型 `OpenAI/OpenViking API KEY`，供问答模块与摘要理解功能调用。

## 3. 一键启动流程

为了免除终端命令调用困难，我们按照不同操作平台分别打包好了：
- **对于 Windows 环境**：直接双击文件夹中的 `start.bat`。（或者在终端运行 `.\start.bat`）
- **对于 Linux / macOS**：在终端运行 `./start.sh`（如提示无权限可以先 `chmod +x start.sh`）。

执行即代表系统执行 `docker compose up -d --build` 的完整链条：
1. 检测底层 Docker 运行环境。
2. 验证 `.env` 文件完整性。
3. 对全套 3 个 Docker 镜像执行自动编译、拉取，并以后台挂载（detatched mode）静默开机。

**待终端提示【成功】后，您可以直接访问：**
- **知识图谱检索系统 (前端)**: [http://localhost](http://localhost)
- **FastAPI / 后端调试后台**: [http://localhost:8000/docs](http://localhost:8000/docs)
- **Neo4j 管控原生界面**: [http://localhost:7474](http://localhost:7474)

## 4. 本地文件怎样同步导入？

本架构遵从**最低代码入侵原则**，保留了基于目录结构做批处理导入的设计：
本工程的 `docker-compose.yml` 已经显式使用 Volumes（卷）将您**宿主机目录** 和 **Docker 应用目录** 的路径进行了同步：

> *本地路径上的 `./output` 和 `./ontology`  <=>  后端容器中的 `/app/output` 和 `/app/ontology` 实现了完全镜像挂载。*

**操作逻辑很简单：**
1. 只需要把生成的 `*.md` 或提炼的 `extracted_triples.json` 放在宿主机原本应该在的位置（根目录下的 `output` 文件夹内）。
2. 在通过 `start.bat` 或者使用命令将该容器重启 / 开机时，后端会在执行正式服务 (`python main.py`) 前**自动检查**并执行 `python init_db.py`。
3. `IngestionService` 因为带有文本校验和摘要比对，它**自动侦测并只把没见过的新文件安全地切制入新库**：
   - 包含通过 Neo4j 追加新的 RAG Graph 的关系。
   - 包含将 Embedding 送往新搭建的环境存储卷 `chroma_db_dir` 以作常驻缓存。

## 5. 常规监控和停盘管理

当您交付完毕或者想释放宿主机资源时，可以通过终端执行此命令安全释放所有微服务进程，但这**完全不会删去**那些已经挂载的包含 `Neo4j` 元数据以及 `Chroma` 库里的数据资产：
```bash
docker compose down
```

如果系统出现非预期错误想要排查某个容器的运行详细信息，可使用：
```bash
# 查看所有容器的复合启动运行日志
docker compose logs -f
```
```bash
# 指定查询后台微服务的报错情况
docker compose logs -f backend
```
